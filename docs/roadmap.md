# Gestalt 로드맵

이 문서는 Gestalt가 다음 단계로 나아가기 위한 방향성을 정리한다. 코드 품질은 이미 상위권(타입 안전성·테스트·CI 모두 양호)이므로, 여기서는 "무엇을 더 만들까"가 아니라 **"무엇을 증명하고, 무엇으로 좁히고, 무엇으로 해자를 팔까"**를 다룬다.

한 문장 요약: **기술적으로는 상위권, 전략적으로는 갈림길.** 기능을 늘려 완결성을 높이는 길보다, Interview·Spec·Memory로 좁혀 해자를 파는 길을 택한다. 그 전에 "해상도 0.8이 정말 효과 있다"를 데이터로 증명하는 것이 모든 작업의 선결 조건이다.

---

## 핵심 진단

차별점은 코드 실행이 아니라 **코드를 쓰기 전, 요구사항의 해상도를 정량화하는 것**이다. 코드 실행은 Claude Code·Cursor·Codex가 이미 잘하고 앞으로 더 잘한다. Gestalt가 비집고 들어갈 자리는 인터뷰와 스펙이다.

그런데 현재 구조에는 세 가지 긴장이 있다.

- **증거의 부재** — "해상도를 올리면 드리프트가 준다"는 핵심 약속을 데이터로 보여주지 못한다.
- **기능 과잉** — 차별점이 아닌 영역(Execute·Code Graph 등)이 전체 LOC의 37%를 차지하며 호스트 기능과 중복된다.
- **얇은 해자** — 복제 불가능한 유일 자산인 Memory가 plain JSON 배열 수준에 머물러 있다.

아래 다섯 갈래와 기술 부채 한 갈래로 나눠 각각의 해결 방향을 정리한다.

---

## 1. 효과 증명 — "0.8이 정말 효과 있다"

### 문제

현재 0.8은 LLM이 스스로 매긴 점수가 임계치를 넘었다는 의미일 뿐이다. 그 점수가 재현되는지, 실제 결과를 개선하는지에 대한 증거가 없다. 제품의 핵심 약속인데 검증 장치가 없다.

### 해결 방향 — 3단계 증명 사다리

| 단계 | 증명할 것 | 방법 |
|------|-----------|------|
| ① 재현성 | 같은 입력 → 같은 점수 | 모호한 요구사항 20개 고정 세트, 각 5회 반복, 점수 분산(σ) 측정 |
| ② 단조성 | 점수가 오르면 결과 품질도 오른다 | 같은 과제를 해상도 0.5 / 0.65 / 0.8 스펙으로 실행하고 산출물 채점 |
| ③ 인과성 | Gestalt를 쓸 때와 안 쓸 때의 차이 | 같은 과제를 "바로 구현" vs "인터뷰 후 구현"으로 나눠 드리프트·재작업률 비교 |

### 첫 액션

`benchmarks/`에 이미 인프라가 있다. 고정 요구사항 세트(golden set) 20개와 평가 루브릭부터 만든다. 채점은 LLM-as-judge로 자동화하되, 5개는 사람이 직접 라벨링해 judge를 보정한다.

### 성공 기준

"해상도 0.8 스펙은 0.5 스펙 대비 재작업률 X% 감소"라는 한 문장을 README 최상단에 박을 수 있으면 성공이다. 이 문장은 마케팅 자산이자 회귀 테스트가 된다.

> ⚠️ ③이 "효과 없음"으로 나올 수도 있다. 그래도 빨리 아는 편이 낫다. 지금 전체 제품이 이 가정 위에 서 있다.

## 2. 게슈탈트 메타포 — 두 레이어로 분리

### 문제

5원리가 전체를 관통하는 일관성은 이 프로젝트의 정체성이자 미학이다. 동시에 신규 사용자에게는 인지 부담이다. "왜 goalClarity가 Closure인가"는 내부적으로는 우아하지만, 처음 쓰는 사람이 배울 이유가 없는 어휘다. 우아함이 진입장벽으로 작동할 위험이 있다.

### 해결 방향 — 표면과 심층 분리

- **표면(사용자 노출)**: "목표 명확도 0.4 → 0.8, 미해결 모순 2건"처럼 평범한 언어
- **심층(아키텍처·문서·브랜딩)**: 게슈탈트 원리를 그대로 유지
- 둘 사이는 매핑 테이블 하나로 연결한다. README에 이미 있는 표가 그 씨앗이다.

### 첫 액션

인터뷰 응답에서 사용자에게 보이는 필드(`label`, `improvementHint`)를 게슈탈트 용어 없이 다시 쓴다. 코드 내부 식별자(`gestaltPrinciple`)는 건드리지 않는다. 표면 문자열만 교체하므로 리스크가 낮고 효과는 즉시 나타난다.

### 성공 기준

게슈탈트를 한 번도 들어본 적 없는 사람이 인터뷰 출력만 보고 "다음에 뭘 명확히 해야 하는지" 이해하면 성공이다. 메타포는 "더 알고 싶은 사람을 위한 깊이"로 내려간다.

## 3. 기능 과잉 — 핵심으로 좁히기

### 문제

차별점이 아닌 영역이 호스트 기능과 중복되며 유지보수 부담만 키운다. LOC 분포가 이를 보여준다. `execute` 4,546줄 + `code-graph` 3,040줄로 전체의 37%가 차별점 바깥에 있다. 정작 차별점인 `interview`는 1,254줄이다.

### 해결 방향 — 기능을 3구역으로 분류

| 구역 | 정의 | 해당 기능 | 방침 |
|------|------|-----------|------|
| **Core (사수)** | 우리만의 차별점 | Interview, Spec, Resolution Scorer, Memory | 투자 집중 |
| **Bridge (얇게)** | 차별점과 실행을 잇는 상태 관리 | Execute 상태머신, Evaluate·Evolve 루프 | 로직 최소화, 실행은 호스트에 위임 |
| **Commodity (재검토)** | 호스트·외부 도구와 중복 | Code Graph, Role Agent 13개, Graph Viz, KB 검색 | "정말 우리가 들고 있어야 하나" 개별 판단 |

### 첫 액션

바로 삭제하지 않는다. 사용 데이터부터 모은다. EventStore에 이벤트가 쌓이고 있으니 각 기능의 실제 호출 빈도를 집계한다. 안 쓰는 기능부터 deprecate 후보로 둔다. 데이터 없이 애착으로 자르면 반발과 후회가 생긴다.

### 성공 기준

"Gestalt는 X를 한다"를 한 문장으로 말할 수 있게 되면 성공이다. 지금은 그 문장이 너무 길다.

## 4. Memory — 진짜 해자 키우기

### 문제

Passthrough 모드(API 키 불필요)는 진입장벽을 낮추는 좋은 선택이다. 동시에 이 선택은 Gestalt를 "상태 관리자 + 프롬프트 생성기"로 만든다. 프롬프트와 게슈탈트 프레임워크는 복제할 수 있다. 복제 불가능한 자산은 누적된 Memory 하나뿐인데, 지금은 plain JSON 배열로 너무 얇다.

### 해결 방향 — "검색 가능한 결정 이력"으로 격상

1. **연결** — KB의 임베딩 인프라를 Memory에 붙여 "예전 비슷한 스펙"을 시맨틱 검색한다. 인프라를 재사용하므로 신규 개발이 적다.
2. **심화** — 결정의 이유·결과·회고까지 스키마를 확장한다. `architectureDecisions: string[]`을 구조화된 레코드로 바꾼다.
3. **공유** — 팀 단위 누적을 강화하고, commit 시 충돌 머지 전략을 정의한다.

### 첫 액션

②부터 시작한다. `architectureDecisions`를 `{ decision, rationale, outcome?, specId, timestamp }` 구조로 바꾼다. 인터뷰 주입 시 "과거 결정과 그 결과"를 함께 넣어 "인터뷰가 점점 똑똑해진다"를 실체화한다.

### 성공 기준

같은 팀이 두 번째 비슷한 과제를 할 때, 인터뷰 라운드 수가 측정 가능하게 줄어들면 성공이다. 1번에서 만든 벤치마크로 측정할 수 있다.

> 📝 3번에서 "재검토" 대상으로 둔 KB 검색 인프라가 여기서 핵심 부품으로 재활용된다. 버리는 게 아니라 핵심으로 끌어오는 것이다.

## 5. 플랫폼 종속 리스크

### 문제

Claude Code에 강하게 결합돼 있다. 좋은 단기 전략이지만, Claude Code가 plan mode·skill·subagent를 계속 강화 중이다. 호스트가 "요구사항 명확화"를 기본 탑재하면 차별점이 증발한다.

### 해결 방향 — "세션"이 아닌 "프로젝트 역사"로 포지셔닝 이동

- 호스트가 잘하는 것: 지금 이 세션의 추론과 실행
- Gestalt가 잡을 것: 프로젝트 전체에 걸친 요구사항의 역사와 일관성 (다시 Memory로 귀결)
- 전술: MCP 표준에만 의존하고, 호스트 전용 기능 의존을 최소화한다. Codex·Gemini 지원은 이미 옳은 방향이다.

### 첫 액션

호스트별 기능 의존을 추상화 레이어 하나로 격리한다. 지금의 `client` 설정(claude-code / codex / both)이 그 씨앗이다. 여기에 호스트 차이를 모두 흡수시켜 코어가 호스트를 모르게 만든다.

### 성공 기준

새 호스트를 추가하는 일이 "어댑터 한 개 작성"으로 끝나면 성공이다. 코어 코드에 호스트 분기가 없어야 한다.

## 기술 부채

방향성 작업을 떠받치는 토대다. 우선순위는 방향성보다 낮지만, `core/` 테스트는 모든 변경의 안전망이라 먼저 처리한다.

| 항목 | 방향 | 트리거 시점 |
|------|------|-------------|
| `core/` 테스트 0% | Result 모나드·config 로딩부터 테스트한다. 모든 것의 기반인데 테스트가 없다 | 지금 |
| 거대 파일 분해 | `evolution.ts`(583줄), `server.ts`(665줄), `code-graph/engine.ts`(562줄)를 해당 기능을 손댈 때 함께 쪼갠다 | 3번 진행 중 |
| KB 검색 O(n) | `search.ts`가 전체 엔트리를 선형 스캔한다. 4번에서 Memory에 붙일 때 함께 개선한다 | 4번과 동시 |
| 프롬프트·파일 IO 추상화 | `prompts.ts`(636줄), `readFileSync` 19개 파일 산재. `PromptBuilder`·`FileStore`로 통합한다 | 후순위 |

> ⚠️ `core/` 테스트 부재는 모순적이다. 자주 바뀌는 `execute`는 123% 커버리지인데, 잘 안 바뀌지만 모두가 의존하는 `core`는 0%다.

## 실행 순서

순서가 중요하다. **증명 → 좁히기 → 키우기** 흐름이다.

```
[0] core/ 테스트 보강 ─────────────── 안전망 (모든 작업의 전제)
      │
[1] 효과 벤치마크 (golden set) ─────── "무엇이 효과 있는지" 측정 도구 확보
      │         │
      │         └──> 이 측정 도구가 3·4번의 판단 근거가 된다
      ▼
[3] 기능 사용 데이터 집계 ──────────── "무엇을 줄일지" 데이터 확보
      │
      ▼
[2] 메타포 표면 분리 ──────────────── 저리스크·즉효, 병렬 진행 가능
      │
      ▼
[4] Memory 격상 (KB 인프라 재활용) ─── 진짜 해자 구축
      │
      ▼
[5] 호스트 추상화 ─────────────────── 방어선
```

### 세 가지 핵심 통찰

1. **벤치마크(1)가 모든 판단의 기준점이다.** 이것 없이 3·4번을 하면 감으로 하는 셈이다.
2. **메타포 분리(2)는 리스크가 가장 낮고 효과는 즉시 나타난다.** 언제든 병렬로 끼울 수 있어 워밍업으로 좋다.
3. **3번에서 줄이려던 KB 인프라가 4번의 핵심 부품이 된다.** 축소와 강화가 한 흐름이다.
