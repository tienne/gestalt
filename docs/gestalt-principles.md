# Gestalt 5원리 — 요구사항 명확화에의 적용

> "전체는 부분의 합보다 크다" — 흩어진 요구사항 조각들을 모아 완전한 스펙으로 결정화한다.

## 개요

게슈탈트 심리학의 5가지 지각 원리를 **요구사항 수집(Interview)**과 **실행 계획(Execute)** 두 단계에 매핑한다. 각 원리는 서로 다른 관점에서 모호성을 줄이는 역할을 한다.

---

## 1. Closure (폐쇄성)

> 불완전한 정보를 완성하려는 경향

### 심리학 원리

사람은 열린 원을 보면 무의식적으로 닫힌 원으로 인식한다. 빈 부분을 채워서 완전한 형태를 만들려는 본능.

### 인터뷰 단계에서

**명시되지 않은 암묵적 요구사항을 찾아낸다.**

질문 전략: _"X를 언급하셨는데, Y는 어떻게 처리해야 하나요?"_

```
사용자: "로그인 기능이 필요해요"
Closure 질문: "로그인 실패 시 처리는 어떻게 하나요?
              비밀번호 찾기도 필요한가요?
              소셜 로그인은요?"
```

→ 사용자가 말하지 않은 **빈 곳**을 찾아 채운다.

### 실행 계획 단계에서

각 완료 조건(Acceptance Criteria)을 **원자적이고 독립적으로 실행 가능한 태스크로 분해**한다. 명시된 완료 조건에서 암묵적으로 필요한 하위 작업까지 식별.

### 가중치

| 프로젝트 유형 | 가중치 |
|:---:|:---:|
| Greenfield | **0.40** (최고) |
| Brownfield | 0.30 |

새 프로젝트일수록 빠진 부분이 많으므로 가중치가 높다.

---

## 2. Proximity (근접성)

> 가까이 있는 요소를 하나의 그룹으로 인식하는 경향

### 심리학 원리

점 6개가 있을 때, 가까이 있는 3개씩을 자연스럽게 두 그룹으로 묶어 인식한다.

### 인터뷰 단계에서

**관련된 요구사항끼리 자연스럽게 묶는다.**

질문 전략: _"A와 B를 하나의 기능으로 합쳐야 할까요?"_

```
사용자가 말한 것들:
  - "사용자 프로필 수정"
  - "프로필 사진 업로드"
  - "비밀번호 변경"
  - "알림 설정"

Proximity 그룹핑:
  [프로필 관리] → 프로필 수정, 사진 업로드
  [계정 설정]  → 비밀번호 변경, 알림 설정
```

→ 같은 **도메인/영역**에 속하는 것끼리 묶는다.

### 실행 계획 단계에서

관련된 원자 태스크를 **도메인 또는 기능 영역별로 그룹핑**한다. 자연스럽게 함께 속하는 태스크들이 하나의 그룹을 이룬다.

### 가중치

| 프로젝트 유형 | 가중치 |
|:---:|:---:|
| Greenfield | 0.25 |
| Brownfield | 0.20 |

---

## 3. Similarity (유사성)

> 비슷한 요소를 같은 그룹으로 인식하는 경향

### 심리학 원리

빨간 원과 파란 원이 섞여 있으면, 색이 같은 것끼리 그룹으로 인식한다. 형태가 아니라 **속성의 유사함**이 기준.

### 인터뷰 단계에서

**요구사항 간 반복되는 패턴을 식별한다.**

질문 전략: _"X와 Y가 같은 패턴을 공유하는데, 일관된 접근 방식을 사용해야 할까요?"_

```
서로 다른 영역의 답변들:
  - "상품 삭제는 관리자만 가능"
  - "사용자 차단도 관리자만"
  - "설정 변경도 관리자 권한 필요"

Similarity 발견:
  → "관리자 권한 체크"라는 공통 패턴이 반복됨
  → 권한 시스템을 공통 모듈로 설계해야 함
```

→ **도메인은 다르지만** 반복되는 패턴을 발견한다.

### Proximity와의 차이

| | Proximity | Similarity |
|---|---|---|
| 묶는 기준 | **같은 영역**에 있어서 | **같은 패턴**이 반복돼서 |
| 결과 | 기능 그룹핑 | 공통 패턴/추상화 발견 |
| 비유 | 같은 서랍에 넣기 | 서랍은 다르지만 똑같이 생긴 물건 발견 |
| 예시 | 프로필 수정 + 사진 업로드 = 프로필 관리 | 상품 삭제 권한 + 사용자 차단 권한 = 권한 시스템 |

### 가중치

| 프로젝트 유형 | 가중치 |
|:---:|:---:|
| Greenfield | 0.20 |
| Brownfield | 0.15 |

---

## 4. Figure-Ground (전경 / 배경)

> 핵심(전경)과 부수적 요소(배경)를 분리하는 경향

### 심리학 원리

루빈의 꽃병 착시 — 같은 이미지에서 꽃병(전경)을 볼 수도, 두 얼굴(배경)을 볼 수도 있다. 무엇을 전경으로 놓느냐에 따라 인식이 달라진다.

### 인터뷰 단계에서

**MVP 범위와 선택사항을 구분한다.**

질문 전략: _"MVP에 반드시 포함되어야 하는 것은 무엇인가요?"_

```
사용자의 요구사항 목록:
  - 로그인/회원가입
  - 상품 검색
  - 장바구니
  - 위시리스트
  - 소셜 공유
  - 리뷰 시스템

Figure-Ground 분류:
  [Figure - MVP] → 로그인, 상품 검색, 장바구니
  [Ground - 후순위] → 위시리스트, 소셜 공유, 리뷰
```

→ 반드시 해야 하는 것(Figure)과 있으면 좋은 것(Ground)을 **명확히 분리**한다.

### 실행 계획 단계에서

완료 조건(AC)을 **essential(figure)과 supplementary(ground)로 분류**하고, 영향도와 의존성 기반으로 우선순위(critical/high/medium/low)를 부여한다.

### 가중치

| 프로젝트 유형 | 가중치 |
|:---:|:---:|
| Greenfield | 0.15 |
| Brownfield | 0.15 |

인터뷰 후반(9라운드 이후)에 기본 적용되며, 스코프 결정의 마지막 관문 역할.

## 5. Continuity (연속성)

> 요소들이 일관된 방향으로 이어진다고 인식하는 경향

### 심리학 원리

교차하는 두 선이 있으면 각각의 선이 끊기지 않고 연속된다고 인식한다. 흐름이 자연스럽게 이어진다는 기대.

### 인터뷰 단계에서

**요구사항 간 모순을 교차 검증한다.**

질문 전략: _"앞서 X라고 하셨는데, 방금 Y라고 하신 건 모순되는 것 아닌가요?"_

```
라운드 2: "모든 데이터는 실시간으로 동기화되어야 해요"
라운드 5: "오프라인에서도 동작해야 해요"

Continuity 감지:
  → "실시간 동기화"와 "오프라인 동작"은 모순
  → 충돌 해결을 위한 추가 질문 필요
     "오프라인에서 수정한 내용이 온라인 복귀 시 충돌하면 어떻게 할까요?"
```

→ 인터뷰 전체 흐름에서 **일관성이 깨지는 지점**을 잡아낸다.

### 실행 계획 단계에서

의존성 그래프(DAG)의 **일관성을 검증**한다. 순환 의존, 그룹 간 충돌이 없는지 확인하고, 위상 정렬 순서와 크리티컬 패스를 도출.

### 특수 규칙

- **모순이 감지되면 다른 모든 원리보다 우선 적용** (override)
- 모순 발견 시 모호성 점수에 **페널티 부여** (0.05 ~ 0.15)

### 가중치

| 프로젝트 유형 | 가중치 |
|:---:|:---:|
| Greenfield | 0 (모순이 발생할 기존 맥락이 없음) |
| Brownfield | **0.20** (기존 시스템과의 일관성이 중요) |

## 원리 적용 순서

### 인터뷰 단계 (라운드 기반)

```
라운드 1-3  [초기]  → Closure 우선 (목표 정의)
라운드 4-8  [중기]  → Proximity ↔ Similarity 교대 (구조화)
라운드 9+   [후기]  → Figure-Ground (우선순위 결정)

※ 모순 감지 시 → 언제든 Continuity가 즉시 개입
※ 모호성이 가장 낮은 차원 → 동적으로 해당 원리 우선 적용
```

### 실행 계획 단계 (고정 순서)

```
Step 1. Figure-Ground  → 완료 조건(AC)을 figure/ground로 분류
Step 2. Closure        → 원자 태스크 분해 (암묵적 태스크 포함)
Step 3. Proximity      → 태스크 그룹핑
Step 4. Continuity     → DAG 검증 (순환/충돌 확인)
```

## 동적 원리 선택 알고리즘

`selectNextPrinciple()`은 매 라운드마다 다음 원리를 동적으로 결정한다.

### 우선순위 체계

```
1. Continuity Override  — 모순이 감지되었으면 무조건 Continuity
        ↓ (모순 없음)
2. Weakest Dimension    — clarity < 0.5인 차원 중 영향도가 가장 큰 원리
        ↓ (모든 차원 ≥ 0.5)
3. Phase Default        — 라운드 번호 기반 기본값
```

### 가장 약한 차원 찾기 (findWeakestDimension)

단순 clarity 비교가 아니라 **가중 영향도**를 계산한다:

```
impact = (1 - clarity) × weight
```

예시: goalClarity=0.4 (weight 0.40) → impact 0.24, constraintClarity=0.3 (weight 0.25) → impact 0.175
→ goalClarity의 영향도가 더 크므로 Closure 선택

이 방식으로 가중치가 높은 차원의 낮은 명확도가 가중치가 낮은 차원보다 우선시된다.

### 라운드 기반 기본값

| 라운드 | 페이즈 | 기본 원리 | 이유 |
|:---:|:---:|---|---|
| 1–3 | early (goal definition) | Closure | 목표부터 완성 |
| 4 | mid (structuring) | Proximity | 구조화 시작 |
| 5 | mid | Similarity | 패턴 식별 |
| 6 | mid | Proximity | 구조화 심화 |
| 7 | mid | Similarity | 패턴 심화 |
| 8 | mid | Proximity | 구조화 마무리 |
| 9+ | late (prioritization) | Figure-Ground | 우선순위 결정 |

> 4–8 라운드에서 Proximity와 Similarity가 **홀/짝 번호로 교대** 적용된다.

## 모호성 점수 (Ambiguity Score)

각 원리는 모호성 점수의 특정 차원과 매핑된다:

| 원리 | 모호성 차원 | 의미 |
|---|---|---|
| Closure | goalClarity | 목표가 얼마나 완전한가 |
| Proximity | constraintClarity | 제약사항이 얼마나 잘 구조화되었나 |
| Similarity | successCriteria | 성공 기준의 패턴이 일관적인가 |
| Figure-Ground | priorityClarity | 우선순위가 명확한가 |
| Continuity | contextClarity | 전체 맥락에 모순이 없는가 (Brownfield 전용) |

### 점수 계산 공식

```
overall = clamp(1.0 - Σ(clarity_i × weight_i) + continuityPenalty)
```

- `clarity_i`: 각 차원의 명확도 (0~1, LLM이 평가)
- `weight_i`: 프로젝트 유형별 가중치 (위 원리별 섹션 참조)
- `continuityPenalty`: 모순 기반 페널티
- `clamp`: 결과를 0~1 범위로 제한

**예시 (Greenfield, 모순 없음):**

```
goalClarity       = 0.8 × 0.40 = 0.32
constraintClarity = 0.7 × 0.25 = 0.175
successCriteria   = 0.6 × 0.20 = 0.12
priorityClarity   = 0.5 × 0.15 = 0.075

overall = 1.0 - (0.32 + 0.175 + 0.12 + 0.075) + 0
        = 1.0 - 0.69
        = 0.31  → 아직 불충분 (threshold 0.2 초과)
```

### Continuity 페널티

인터뷰 중 모순이 감지되면 모호성 점수에 페널티가 추가된다:

```
모순 0개 → 페널티 0
모순 1개 → 0.05 + (0.10 × 1/3) ≈ 0.083
모순 2개 → 0.05 + (0.10 × 2/3) ≈ 0.117
모순 3개+ → 0.15 (최대)
```

공식: `penalty = 0.05 + 0.10 × min(contradictions / 3, 1)`

모순이 해소되지 않으면 모호성 점수가 threshold(0.2) 아래로 내려가기 어렵다. 이는 모순 해결을 강제하는 메커니즘이다.

### 종료 조건

**모호성 점수가 0.2 이하**가 되면 요구사항이 충분히 명확하다고 판단하고 인터뷰를 종료할 수 있다. (`isReady = overall ≤ 0.2`)
