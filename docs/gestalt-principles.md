# 게슈탈트 원리 — 요구사항 명확화에 적용한 이유

> "전체는 부분의 합보다 크다"

---

## 왜 이 접근법이 필요했나

소프트웨어 프로젝트에서 요구사항 수집이 실패하는 이유는 대부분 질문을 안 해서가 아니에요. 잘못된 순서로, 잘못된 질문을 해서예요.

"무엇을 만들고 싶으세요?" 한 방으로 전체 요구사항을 끌어내려 하거나, 반대로 너무 자잘한 것부터 물어보다가 핵심을 놓쳐요. 맥락 없이 모인 정보 조각들은 그 자체로는 실행 계획이 되지 않아요.

게슈탈트 심리학은 1910년대부터 인간이 흩어진 요소들을 어떻게 하나의 의미 있는 전체로 인식하는지를 연구해왔어요. 핵심 주장은 단순해요 — 인식은 요소의 합이 아니라 요소들 사이의 **관계**에서 나온다는 거예요. Gestalt는 이 원리를 요구사항 수집에 그대로 가져왔어요. 흩어진 요구사항 조각들을 관계와 구조로 엮어서 완전한 스펙으로 만들어요.

---

## 5가지 원리와 요구사항의 대응

각 원리는 요구사항의 서로 다른 빈 곳을 채워요. 원리 하나가 답하는 질문이 다르기 때문에 순서와 타이밍이 중요해요.

---

### 1. Closure (폐쇄성) — 빈 곳을 찾아내는 원리

**심리학 원리:** 열린 원을 보면 뇌는 자동으로 닫힌 원으로 인식해요. 불완전한 정보를 완성하려는 본능이에요.

**요구사항에서 하는 일:** 사용자가 말하지 않은 암묵적 요구사항을 찾아내요. "로그인이 필요해요"라는 말 뒤에는 로그인 실패 처리, 비밀번호 찾기, 세션 만료 정책이 숨어 있어요. Closure는 그 빈 곳을 드러내는 질문을 해요.

```
사용자: "로그인 기능이 필요해요"
Closure 질문:
  "로그인 실패 시 처리는 어떻게 하나요?"
  "비밀번호 찾기도 필요한가요?"
  "소셜 로그인은 고려하고 있나요?"
```

**실행 계획에서 하는 일:** 완료 조건(Acceptance Criteria)을 원자적이고 독립적으로 실행 가능한 태스크로 쪼개요. 명시된 조건뿐 아니라 암묵적으로 필요한 하위 작업까지 포함해요.

**해상도 차원:** `goalClarity` — 목표가 얼마나 완전하게 정의되었는가

| 프로젝트 유형 | 가중치 |
|:---:|:---:|
| Greenfield (새 프로젝트) | **0.40** |
| Brownfield (기존 시스템) | 0.30 |

새 프로젝트일수록 채워야 할 빈 곳이 많아서 가중치가 높아요.

---

### 2. Proximity (근접성) — 관련된 것끼리 묶는 원리

**심리학 원리:** 점 6개가 있을 때 가까이 있는 3개씩을 두 그룹으로 자연스럽게 묶어 인식해요. 거리가 그룹을 결정해요.

**요구사항에서 하는 일:** 같은 도메인이나 영역에 속하는 요구사항끼리 자연스럽게 묶어요.

```
사용자가 언급한 것들:
  - "사용자 프로필 수정"
  - "프로필 사진 업로드"
  - "비밀번호 변경"
  - "알림 설정"

Proximity 그룹핑:
  [프로필 관리] → 프로필 수정, 사진 업로드
  [계정 설정]  → 비밀번호 변경, 알림 설정
```

"A와 B를 하나의 기능으로 합쳐도 될까요?" 같은 질문으로 사용자의 멘탈 모델을 확인해요.

**실행 계획에서 하는 일:** 원자 태스크를 도메인 또는 기능 영역별로 그룹핑해요. 함께 속하는 태스크들이 하나의 그룹을 이루도록 구성해요.

**해상도 차원:** `constraintClarity` — 제약사항이 얼마나 잘 구조화되었는가

| 프로젝트 유형 | 가중치 |
|:---:|:---:|
| Greenfield | 0.25 |
| Brownfield | 0.20 |

---

### 3. Similarity (유사성) — 패턴을 발견하는 원리

**심리학 원리:** 빨간 원과 파란 원이 섞여 있으면 형태보다 색이 같은 것끼리 그룹으로 인식해요. 위치가 아니라 속성의 유사함이 기준이에요.

**요구사항에서 하는 일:** 서로 다른 영역의 요구사항에서 반복되는 패턴을 찾아내요.

```
서로 다른 영역의 요구사항들:
  - "상품 삭제는 관리자만 가능"
  - "사용자 차단도 관리자만"
  - "설정 변경도 관리자 권한 필요"

Similarity 발견:
  → "관리자 권한 체크"가 반복됨
  → 권한 시스템을 공통 모듈로 설계해야 함
```

"X와 Y가 같은 패턴을 공유하는데, 일관된 방식을 써야 할까요?" 같은 질문을 해요.

**Proximity와의 차이:**

Proximity는 "같은 서랍에 넣기"예요 — 같은 영역이라서 묶어요.
Similarity는 "다른 서랍에 있지만 똑같이 생긴 물건 발견"이에요 — 같은 패턴이라서 발견해요.

| | Proximity | Similarity |
|---|---|---|
| 묶는 기준 | 같은 영역에 있어서 | 같은 패턴이 반복돼서 |
| 결과 | 기능 그룹핑 | 공통 추상화 발견 |
| 예시 | 프로필 수정 + 사진 업로드 = 프로필 관리 | 상품 삭제 권한 + 사용자 차단 권한 = 권한 시스템 |

**해상도 차원:** `successCriteria` — 성공 기준의 패턴이 일관적인가

| 프로젝트 유형 | 가중치 |
|:---:|:---:|
| Greenfield | 0.20 |
| Brownfield | 0.15 |

---

### 4. Figure-Ground (전경/배경) — 핵심과 선택사항을 구분하는 원리

**심리학 원리:** 루빈의 꽃병 착시가 대표적인 예예요. 같은 이미지에서 꽃병(전경)을 볼 수도, 두 얼굴(배경)을 볼 수도 있어요. 무엇을 전경으로 놓느냐에 따라 전혀 다른 것을 보게 돼요.

**요구사항에서 하는 일:** MVP 범위(Figure)와 선택사항(Ground)을 명확히 분리해요. "MVP에 반드시 포함되어야 하는 것은 무엇인가요?" 같은 질문으로 우선순위를 끌어내요.

```
전체 요구사항 목록:
  로그인/회원가입, 상품 검색, 장바구니,
  위시리스트, 소셜 공유, 리뷰 시스템

Figure-Ground 분류:
  [Figure — MVP]   → 로그인, 상품 검색, 장바구니
  [Ground — 후순위] → 위시리스트, 소셜 공유, 리뷰
```

**실행 계획에서 하는 일:** 완료 조건을 `essential`(figure)과 `supplementary`(ground)로 분류하고, 영향도와 의존성을 기반으로 `critical / high / medium / low` 우선순위를 부여해요.

**해상도 차원:** `priorityClarity` — 우선순위가 명확한가

| 프로젝트 유형 | 가중치 |
|:---:|:---:|
| Greenfield | 0.15 |
| Brownfield | 0.15 |

인터뷰 후반(9라운드 이후)에 기본 적용되며, 스코프 결정의 마지막 관문 역할을 해요.

---

### 5. Continuity (연속성) — 모순을 잡아내는 원리

**심리학 원리:** 교차하는 두 선이 있으면 각각의 선이 끊기지 않고 이어진다고 인식해요. 흐름이 자연스럽게 연속된다는 기대에서 나온 원리예요.

**요구사항에서 하는 일:** 인터뷰 전체 흐름에서 일관성이 깨지는 지점을 잡아내요.

```
2라운드: "모든 데이터는 실시간으로 동기화되어야 해요"
5라운드: "오프라인에서도 동작해야 해요"

Continuity 감지:
  → "실시간 동기화"와 "오프라인 동작"은 서로 충돌
  → 추가 질문: "오프라인에서 수정한 내용이 온라인 복귀 시
     충돌하면 어떻게 처리할까요?"
```

**실행 계획에서 하는 일:** 의존성 그래프(DAG)의 일관성을 검증해요. 순환 의존과 그룹 간 충돌을 확인하고, 위상 정렬 순서와 크리티컬 패스를 도출해요.

**특수 규칙:** 모순이 감지되면 다른 모든 원리보다 먼저 개입해요. 모순이 해소되지 않으면 해상도 페널티가 쌓여서 0.8에 도달하기 어렵게 설계돼 있어요.

**해상도 차원:** `contextClarity` — 전체 맥락에 모순이 없는가

| 프로젝트 유형 | 가중치 |
|:---:|:---:|
| Greenfield | 0 (기존 맥락이 없어서 모순이 생기기 어려워요) |
| Brownfield | **0.20** (기존 시스템과의 일관성이 중요해요) |

## 원리 적용 순서

### 인터뷰 단계

라운드 번호에 따라 기본 원리가 달라지지만, 그게 전부가 아니에요. 매 라운드마다 동적으로 원리를 선택해요.

```
라운드 1-3  [초기]  → Closure 우선 (목표 정의)
라운드 4-8  [중기]  → Proximity ↔ Similarity 교대 (구조화)
라운드 9+   [후기]  → Figure-Ground (우선순위 결정)

모순 감지 시 → 언제든 Continuity가 즉시 개입
해상도가 가장 낮은 차원 → 해당 원리를 동적으로 우선 적용
```

### 실행 계획 단계

고정된 순서로 진행돼요.

```
Step 1. Figure-Ground  → 완료 조건을 figure/ground로 분류
Step 2. Closure        → 원자 태스크 분해 (암묵적 태스크 포함)
Step 3. Proximity      → 태스크 그룹핑
Step 4. Continuity     → DAG 검증 (순환/충돌 확인)
```

## 동적 원리 선택 — selectNextPrinciple()

매 라운드마다 어떤 원리를 쓸지 결정하는 알고리즘이에요. 세 가지 우선순위로 작동해요.

```
1. Continuity Override  — 모순이 감지되면 무조건 Continuity
        ↓ (모순 없음)
2. Weakest Dimension    — clarity < 0.5인 차원 중 영향도가 가장 큰 원리
        ↓ (모든 차원 ≥ 0.5)
3. Phase Default        — 라운드 번호 기반 기본값
```

**가장 약한 차원 찾기 (findWeakestDimension):** 단순히 clarity 점수가 낮은 걸 고르는 게 아니에요. 가중 영향도를 계산해요.

```
impact = (1 - clarity) × weight
```

예를 들어 goalClarity=0.4 (weight 0.40)이면 impact 0.24이고, constraintClarity=0.3 (weight 0.25)이면 impact 0.175예요. clarity는 goalClarity보다 낮지만 영향도는 더 작아요. 그래서 Closure를 선택해요.

가중치가 높은 차원이 조금만 불명확해도, 가중치가 낮은 차원이 많이 불명확한 것보다 먼저 다뤄요.

**라운드 기반 기본값:**

| 라운드 | 페이즈 | 기본 원리 | 이유 |
|:---:|:---:|---|---|
| 1–3 | 초기 (목표 정의) | Closure | 목표부터 완성 |
| 4 | 중기 (구조화) | Proximity | 구조화 시작 |
| 5 | 중기 | Similarity | 패턴 식별 |
| 6 | 중기 | Proximity | 구조화 심화 |
| 7 | 중기 | Similarity | 패턴 심화 |
| 8 | 중기 | Proximity | 구조화 마무리 |
| 9+ | 후기 (우선순위) | Figure-Ground | 우선순위 결정 |

4–8라운드에서 Proximity와 Similarity가 홀/짝 번호로 교대해요.

## 해상도 점수 (Resolution Score)

인터뷰가 충분히 진행되었는지 수치로 판단하는 지표예요. 0.8 이상이 되면 요구사항이 실행 계획을 만들기에 충분히 명확하다고 봐요.

**각 원리와 해상도 차원의 대응:**

| 원리 | 해상도 차원 | 의미 |
|---|---|---|
| Closure | goalClarity | 목표가 얼마나 완전한가 |
| Proximity | constraintClarity | 제약사항이 얼마나 잘 구조화되었나 |
| Similarity | successCriteria | 성공 기준의 패턴이 일관적인가 |
| Figure-Ground | priorityClarity | 우선순위가 명확한가 |
| Continuity | contextClarity | 전체 맥락에 모순이 없는가 (Brownfield 전용) |

**점수 계산:**

```
overall = clamp(Σ(clarity_i × weight_i) - continuityPenalty)
```

- `clarity_i`: 각 차원의 명확도 (0~1, LLM이 평가)
- `weight_i`: 프로젝트 유형별 가중치
- `continuityPenalty`: 모순 기반 페널티
- `clamp`: 결과를 0~1 범위로 제한

**예시 (Greenfield, 모순 없음):**

```
goalClarity       = 0.8 × 0.40 = 0.320
constraintClarity = 0.7 × 0.25 = 0.175
successCriteria   = 0.6 × 0.20 = 0.120
priorityClarity   = 0.5 × 0.15 = 0.075

overall = 0.690
→ 아직 부족 (threshold 0.8 미만, 인터뷰 계속)
```

**Continuity 페널티:**

모순이 발견될 때마다 페널티가 쌓여요.

```
모순 0개 → 0
모순 1개 → 0.05 + (0.10 × 1/3) ≈ 0.083
모순 2개 → 0.05 + (0.10 × 2/3) ≈ 0.117
모순 3개+ → 0.15 (최대)
```

공식: `penalty = 0.05 + 0.10 × min(contradictions / 3, 1)`

모순을 해소하지 않으면 해상도가 0.8에 도달하기 어렵게 설계한 이유는, 모순이 남아 있는 요구사항으로 만든 실행 계획은 반드시 문제가 생기기 때문이에요.

**종료 조건:** `overall ≥ 0.8`이면 `isReady = true`, 인터뷰를 종료할 수 있어요.
